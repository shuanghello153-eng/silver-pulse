# 标签 / 搜索 架构重构方案（搜索优先 + 扁平标签池）

> **本稿只出方案、不改数据、不部署。** 核心转变：你推翻了上一版"标签绝不跟分类重名"的铁律——规则服务于目标，目标变了规则就改。

---

## 0. 我的结论（先说人话）

1. **你的方向对**：银发经济是"360 行都能沾边"的行业，硬塞进 13 个一级分类的树里，用户根本不知道某概念挂在哪根枝上（"老年眼镜"埋在"日常用品→最后一级"，没人会去点）。**搜索才是最高效的入口**，这点我完全同意。
2. **我补一点 refinement（不推翻你）**：分类**别删**，留在数据库当"后台维度"——评分的"中心程度"、给另一 AI 打标签、后续分析都还要用它。前台**不摆分类树**，最多给每张卡片一行小面包屑（企业属于哪个分类），成本极低、还能帮用户定位。
3. **标签 = 扁平可搜词池，不要 L1/L2 层级**。你当时也问过"标签到底要不要分一级二级"——在"搜索优先"这个场景里，**扁平比层级好**：搜索是无差别匹配，层级反而增加维护负担。
4. **"标签太多展示不下"的担心，数据打脸了**：合并后每家企业**最多 6 个标签**，1108 家只有 1–3 个，202 家 0 个。**展示毫无压力**，卡片全显示都不挤。

## 2. 新架构：搜索优先，标签当词池，分类退后台

```
┌─ 数据库层（全留，用户不直接爬）──────────────┐
│  分类 13L1/78L2  ← 评分/分析/另一AI用        │
│  标签池 505 种     ← 企业自由关联，可多标签    │
│  同义词表           ← 同类词扩张（搜索解锁关键） │
└───────────────────────────────────────────────┘
            ↓ 搜索时全部拼进索引
┌─ 搜索层（主角）──────────────────────────────┐
│  输入框 → 匹配 名称+描述+标签+同义词扩展      │
│  → 结果列表（扁平、无层级、想怎么搜怎么搜）   │
└───────────────────────────────────────────────┘
            ↓ 展示
┌─ 展示层 ────────────────────────────────────┐
│  卡片：名称 + 顶多3个高亮标签 + 分类小面包屑  │
│  筛选栏：可选、可关，只放~15个高频标签当快捷   │
└───────────────────────────────────────────────┘
```

## 3. 同义词扩展层（你提的"同类词扩张"，这是关键解锁）

**为什么必须有它**：没有同义词，"老年眼镜"就永远埋在"日常用品→老年护肤"里用户找不到。有了同义词表，搜"眼镜/老花镜/老花眼/视觉障碍"都能命中同一批企业——**这正好绕开了"标签要不要和分类重名"的老问题：不用重名，搜索扩展一下就通了。**

## 4. 展示策略（解决"标签太多 / 分类太乱"）

- **卡片**：企业名称 + **顶多 3 个"高亮标签"**（优先挑资本信号"融资/IPO"、或频次最高的）+ 一行**分类小面包屑**（如「日常用品 · 假发」）。
- **筛选栏**：只放 **~15 个最高频 / 最有选题价值**的标签当"快捷筛选"（居家护理、AI科技、消费品、远程医疗、认知症、养老金融、融资/IPO…）。你怀疑筛选有用——我同意，所以做成**可选、可一键关掉**，主力还是搜索。
- **分类树**：前台不摆。留在数据库。

## 5. 清理规则（垃圾清，但不因"和分类重名"而删）

| 仍要清的（垃圾） | 不再删的（之前误删，已拿回） |
|---|---|
| 国家/地区、日期、数据源品牌(Crunchbase/APP)、单字碎片、纯数字 | 和分类同名的标签：老年文娱/假发/专业级护理/保健品&OTC/慢病管理/运营商/认知症/康复/社交平台/临终关怀… 统统留作搜索词 |

**给另一 AI 的打标签规则**：可在 505 池里选词或提新词，但提新词必须带"依据字段"（分类/描述/选题库关联），禁垃圾词；每企业标签上限 6 个（数据证明够用）。

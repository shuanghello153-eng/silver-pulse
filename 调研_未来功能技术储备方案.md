# 银脉（Silver Pulse）· 未来功能技术储备方案

> 编写日期：2026-07-09 ｜ 角色：技术规划（只做方案，不写实现代码）
> 背景：当前已落地 RSS 采集、六维规则评分、事件聚类、企业库（1118 家）、收藏机制（localStorage + feedback.jsonl 回灌）、选题雷达、每日健康报告、0722 降级预案。
> 本文把此前被标为"远期暂缓"的方向**逐一细化成可执行方案**，即使部分最终不做，也先把思路、架构、成本与冲突点想清楚。

**架构前提（所有方案都必须遵守）**
- 站点 = 纯静态（`index.html` / `enterprise.html` / `about.html`），由 Python 脚本本地生成，GitHub Pages 托管，**无后端、无运行时数据库**。

## 1. 企业分自进化（研究价值分随新资讯自动调整闭环）

### 目标
让 `enterprise_score.py` 算出的 `research_value` 成为**活的分数**：每当有新的相关资讯进入，企业分自动抬升；事件冷却后温和回落；长期被反复报道的"高频企业"自然获得更高基准。当前 `event_boost`（7 天 / 30 天窗口，cap=35）只做"当前窗口有没有大事"的瞬时加成，没有记忆、没有衰减曲线，因此不是闭环。

### 可行性评估
**高（P1）**。约 70% 脚手架已存在：`event_boost` 雏形、`compute_event_boost()` 已读取 `scored_latest.json`、`research_value` 已写回 `enterprise_scores.json` 并烘焙进 `enterprise.html`。缺口只在"跨周记忆 + 衰减 + 闭环触发"，全部是纯代码、零模型成本。config.py:861 已注明"企业库数据积累到 2000+ 家后再开启"——但自进化的*机制*现在就可以先建，用 1118 家的数据跑通即可。

### 技术方案 / 架构草图
把"瞬时 event_boost"升级为"**持久化事件记忆 + 指数衰减**"：

```
data/enterprise/event_memory.json   # 新增：跨周持久化的企业事件账本
  { serial: {
      "events": [ {date, event_type, signal_score, url_hash} ],
      "last_boost": float,          # 上次落地的 boost 值
      "baseline_adj": float,        # 长期高频企业的基准上浮（≤ MIRROR_BONUS 量级）
      "updated": "2026-07-09"
  }}

闭环（每次流水线 run_daily）：
  1. collector/scored_latest 出新资讯
  2. enterprise_score.main():
       - 计算"本窗口增量 boost"（沿用现有 7d/30d 规则）
       - 从 event_memory 读取历史事件，对 >30d 的事件按半衰期衰减（如 30d 半衰 → boost*0.5）
       - 累加"未衰减完的历史 boost"，得到 persistent_boost
       - research_value = base + persistent_boost + mirror_bonus（cap 100）
       - 高频企业（近 90d 事件数≥N）→ baseline_adj 微涨（夹紧到 +3 以内）
  3. 把本次结果写回 event_memory（含 decay 后的值），供下一周继续衰减
  4. 写回 enterprise_scores.json → 烘焙 enterprise.html
```

护栏：衰减半衰期、baseline_adj 上限、事件账本保留期都做成 `config.py` 常量；任何自动改动不触碰 `all_enterprises.json` 的其他字段（沿用现有"merge 步骤拥有写回权"的设计，见 `enterprise_score.py:397` 注释）。

### 成本估算
- 算力：读/写一个小 JSON（1118 家 → 约百 KB），可忽略。
- 模型：零。纯规则 + 数学。
- 维护：账本需随企业库清理（删除已移除企业）时同步清理，约 0.5 人日。

### 与无后端静态站架构的冲突点
**无冲突**。全程在 build time 完成，结果烤进 HTML。唯一要注意：分数变化只在每周/每日流水线跑时才体现，用户当天收藏不会立刻改企业分——这是架构固有限制，需在 about 页注明"企业分每周刷新"。

### 建议优先级 & 启动时机
**P1（高）**。机制现在可建、零成本、直接提升网站"活"的感觉。建议：2026-07 内先落地 `event_memory` + 衰减逻辑（仍保留 `ENABLE_AUTO_THRESHOLD=False` 的保守开关），与 0722 降级解耦，独立上线。

## 2. 强模型评分（HY3 免费期结束后用 GPT/Claude 做五维精评）

### 目标
免费期结束后，在"规则兜底已够用"的基础上，**选择性**用强模型（GPT-4o-mini / Claude Haiku 等廉价档）对高分候选做五维精评（industry/signal/writing/cn_fit/urgency/novelty），产出更稳、能读懂长文的评分与可解释理由，替代部分规则盲区（如长文主题误判、反常识度判断）。

### 可行性评估
**中（P2）**。当前 `config.L3_STRONG_MODEL="hunyuan"`（HY3）仅对 5 维打分、绝不自动定终分，且免费期后才有真实成本。落地难点不在代码，而在**密钥管理、缓存、降级三件套**。规则兜底已能跑，所以强模型是"增强"而非"必需"——优先级取决于小爽对评分精度的容忍度。

### 技术方案 / 架构草图
```
A. API Key 管理（绝不进 git）
   - 密钥存于自动化框架的 Secret / 环境变量（如 OPENAI_API_KEY / ANTHROPIC_API_KEY）
   - config.py 仅读 env:  os.environ.get("SP_STRONG_API_KEY")
   - .gitignore 已屏蔽（确认无密钥提交）；本地调试用 .env（不提交）

B. 路由（只对"值得用模型"的条目调用，省钱）
   - 第一道：规则粗筛 final_score≥阈值(如≥4.0) 才进入强模型候选池
   - 第二道：已缓存(url_hash+模型版本 命中) → 直接复用，不调 API
   - 单周强模型调用量 ≈ 精选候选 ~60-120 条，可控

C. 缓存层
   data/model_cache/<model>_<url_hash>.json
     { dims:{...}, reason:"...", model:"gpt-4o-mini", ts, cost_usd }
   - key 含"模型版本号"，模型切换即失效重算
   - 缓存永不过期（同一文章理由不变），大幅降复跑成本

D. 降级策略（强模型不可用时）
   - API 超时 / 5xx / 额度耗尽 → 捕获异常 → 回退规则分值 + 模板理由
   - 当日强模型调用费超预算(如 $0.2) → 抛"成本熔断"→ 剩余条目全回退规则
   - 终分仍由规则权重合成，强模型分仅作"覆盖建议"，不破坏现有铁律
```

### 成本估算（按 ~80 条/周强模型精评估算）
- GPT-4o-mini：~$0.00015/1k input tokens，长文摘要 ~800 tokens → 单条 ~$0.0002，80 条 ≈ **$0.016/周 ≈ $0.7/月**。
- Claude Haiku：略高但同量级，**<$2/月**。
- 含缓存命中后，实际费用再降 50-70%。**结论：强模型增强是"可忽略成本"级别**，真正成本在"全量用强模型"——所以必须路由 + 缓存。

### 与无后端静态站架构的冲突点
**轻微冲突，可解**。强模型调用发生在 build time（流水线运行时），结果烤进 HTML，不违反无后端原则。唯一冲突点：若未来想"用户打开页面时实时精评某条"，则必须引入后端/Serverless——本文方案**刻意不这么做**，保持静态。

### 建议优先级 & 启动时机
**P2（中）**。在 0722 免费期结束前 1-2 周准备（密钥 + 缓存 + 降级代码），但**默认开关保持关闭**，先沿用规则；等小爽实测觉得规则分不够用时再开。建议：2026-07-15 前完成代码储备，0722 当天按 `0722_token_downgrade_plan.md` 决策是否启用。

## 3. B站 / 视频平台受众热度作为次要信号

### 目标
把"某银发话题在 B 站 / 视频平台的受众热度（播放、点赞、投币、弹幕）"作为**次级信号**，加权到选题评分的 `signal` 维度或单独的 `audience_heat` 字段，捕捉"国内年轻人/从业者正在关注什么"，补强当前纯图文 RSS 的盲区。

### 可行性评估
**低-中（P3）**。技术可行（B 站有公开搜索/排行榜接口、RSS 社区源），但三条硬约束：(1) 视频平台信号**更新快、衰减快**，与"周更静态站"天然错配；(2) 合规风险（抓取协议、版权、平台 ToS）；(3) 信号噪声大（标题党、搬运）。建议只做"观察池 + 弱加权"，不进终分主体。

### 技术方案 / 架构草图
```
采集（build time，每周一次）：
  - B 站：search API（关键词 = 银发/养老/老年/适老）+ 按播放排序，取 TOP N
    （或用社区 RSS 如 bilibili-video-rss 规避直接爬虫）
  - 去敏感字段，只存：标题、UP主、播放量、发布日期、视频 BV 号、话题标签
  - 落 data/audience_heat.json（与 scored_latest 无关，独立信号表）

加权（弱、可关）：
  - 计算 topic_heat = log(播放+1) 归一化到 0-5
  - 在聚类后，对"同话题"的资讯打 audience_heat 标签（仅展示，不进 final_score）
  - 或作为 signal 维度的 +0~1 微调（需 config 开关，默认 0）
  - 在 about / 选题雷达页增加"本周视频平台热点"小模块（纯展示）
```

### 成本估算
- 算力：一次搜索 + 解析，可忽略。
- 合规成本：需人工确认接口 ToS / 改用官方开放接口；建议只读公开排行榜，不落视频本体。
- 维护：视频平台接口变动频繁，约 1 人日/季度。

### 与无后端静态站架构的冲突点
**中冲突（时效性）**。视频热度是"天级甚至小时级"信号，但站点周更 → 烘焙进 HTML 时热度已旧 7 天。方案：**只作弱展示信号 + 明确标注"采集于 X 日"**，不承诺实时。若未来要实时，必须 Serverless 代理——本文不纳入。

### 建议优先级 & 启动时机
**P3（低）**。优先级最低，因时效错配 + 合规风险 + 噪声。建议：等企业分自进化、0722 降级、聚类回测都稳了之后（2026-Q4 或之后）再评估；先用"展示模块"试水，不进终分。

## 4. 0722 降级策略（免费期结束后低成本运行：分层路由降本）

### 目标
HY3 免费期（约 2026-07-22）结束后，让整套"采集→评分→聚类→生成→部署"流水线在**近乎零真实成本**下继续跑，核心质量不掉。

### 可行性评估
**高（P1，已有预案）**。`0722_token_downgrade_plan.md` 已实测结论：**当前评分/聚类/推荐全部是规则兜底，不调 LLM**，token 消耗只在"自动化框架读脚本 stdout + 写执行报告"。所以降级空间大，风险低。

### 技术方案 / 架构草图（在现有预案上补"分层路由"细节）
```
现有已落地降本：标签池迭代 每日→每周一；脚本静音（只回一行 JSON 摘要）。
新增"关键词初筛 → 低配模型 → 高配模型"三级路由（针对少数真要调模型的环节）：

  L0 规则兜底（默认，零成本）：评分/聚类/推荐全用 config 权重，不调模型。
  L1 轻量模型（如 hy-lite / hunyuan-lite）：仅"每日摘要推送"套模板环节（读 JSON+排序+模板），不需深度推理。
  L2 强模型（HY3 / GPT）：仅"每日资讯更新"的调度决策（异常判断、部署判定），每日仅 1 次，成本可控。

路由阈值（落到 config）：
  - 摘要推送：永远走 L1（P1 已规划）
  - 标签池迭代：L0（纯脚本，零模型）
  - 资讯更新调度：L2，但加"成本熔断"——当日累计模型费 > $0.05 则后续步骤降级 L0
  - 强模型五维精评（方向2）：默认关，超预算绝对不触发

护栏：所有 modelId 在自动化配置里可一行改回 hy3；0722 后连续 3 天人工抽查推送/部署质量，退化即回滚。
```
（详细执行清单见 `0722_token_downgrade_plan.md`，本文不重复，仅补充分层路由阈值。）

### 成本估算
- 若严格按 L0/L1/L2 路由：**月成本 ≈ $0（免费接口）+ 少量 L2 调度费 <$1/月**。
- 风险场景（误全开 L2）：可能升至 $5-10/月，靠熔断阀避免。

### 与无后端静态站架构的冲突点
**无冲突**。降级完全在流水线运行侧，不影响静态产物。

### 建议优先级 & 启动时机
**P1（高，已规划）**。2026-07-22 当天按预案执行；分层路由阈值代码在 07-15 前补完。

## 5. A/B 影子评分（新模型跑影子、老模型做主，不污染生产）

### 目标
引入新评分模型（更强模型 / 新权重方案）时，**同时跑新旧两套**，新模型只产"影子分"用于对比校验，生产终分仍由老模型决定。用数据证明新模型更准后再切换，避免"一键替换翻车"。

### 可行性评估
**中（P2）**。架构上零冲突——影子分就是多写一个 JSON，不碰 `scored_latest.json`。难点在"怎么公平对比"和"多久才敢切换"。config.py:859 已注明"A/B 影子评分：未启用 → 周模式数据点太少，日模式再考虑"，说明团队已有保留态度。

### 技术方案 / 架构草图
```
流水线新增一步（build time，可选开关 ENABLE_SHADOW_SCORING）：
  1. 主路径不变：老模型/规则 → scored_latest.json（生产，烘焙 HTML）
  2. 影子路径：新模型 → data/scored_shadow.json（独立文件，绝不写回生产）
     - 同一批文章、同一输入，保证可比
  3. 对比报告（data/ab_report.md）：
     - 分差分布（|shadow - main| 的均值/分位）
     - 精选集合差异（哪些进/出精选，召回变化）
     - 与 feedback.jsonl 的 verdict 比对：影子分命中率 vs 主分命中率
  4. 累积 N 周（建议 ≥4 周、≥200 条带反馈样本）后：
     - 若影子分命中率显著更高且分差稳定 → 提"切换建议"给小爽人工确认
     - 绝不自动切换（护栏：遵循"阈值/模型调整需人工确认"铁律）

不污染生产的关键：
  - scored_shadow.json 命名隔离，generator.py 只读 scored_latest.json
  - 影子分字段不进入任何 HTML 模板
  - 影子路径异常 → 静默跳过，不影响主路径
```

### 成本估算
- 算力：双倍跑评分逻辑，规则侧可忽略；若影子用强模型，按方向 2 成本 ×2（仍 <$2/月，且可仅对精选候选跑影子）。
- 人力：每周看一眼 ab_report，约 5 分钟。

### 与无后端静态站架构的冲突点
**无冲突**。影子分是 build time 副产物，不进静态页。

### 建议优先级 & 启动时机
**P2（中）**。建议：方向 2（强模型）启用后、或权重大改前再开影子。时点约 2026-Q3 末。先做"规则新权重 vs 旧权重"的影子（零成本）练手，再上"模型 vs 规则"影子。

## 6. 移动端适配（桌面优先 → 最小成本移动可用）

### 目标
当前 `ui_common.py` / `generator.py` 产出桌面优先布局。让手机用户能基本可用（阅读资讯列表、企业库、收藏），不追求完美，但别出现"横向滚动 / 按钮点不到"。

### 可行性评估
**高（P1）**。纯前端 CSS 问题，零模型、零后端。决策点只有一个：**响应式 CSS vs 单独移动站**。鉴于全站由单一生成器产出，响应式 CSS 明显更优（避免双份模板维护地狱）。

### 技术方案 / 架构草图
```
方案：响应式 CSS（强烈推荐），不动生成器结构，只改 ui_common.py 输出的 <style>：
  - 容器 max-width 自适应；grid 在 <768px 变单列
  - 导航：桌面横向 → 移动端顶部汉堡/折叠（纯 CSS :checked toggle，无 JS 依赖也可）
  - 表格（企业库）改为卡片式或横向滚动容器；分数/理由字号适配
  - 收藏按钮触摸区 ≥44px
  - 用 @media (max-width:768px) 渐进增强，桌面布局零改动

为何不选"单独移动站"：
  - 需维护两套 generator 模板，成本翻倍
  - 静态站无设备路由，要靠 JS 跳转或双 URL，体验差
  - 响应式一次改完，长期维护成本低
```

验证：复用现有 `visual_regression.py`（已存在）做桌面+移动视口截图对比，防止改崩。

### 成本估算
- 一次性：响应式 CSS 改造 + 移动视口回归测试，约 2-3 人日。
- 维护：随模板改动顺带适配，可忽略。

### 与无后端静态站架构的冲突点
**无冲突**。纯静态 CSS/HTML，GitHub Pages 直接服务。

### 建议优先级 & 启动时机
**P1（高）**。建议紧跟"界面地基修复"（项目全景文档指出界面是从没人真正看过导致反复出错）之后做，2026-07 内完成，让小爽手机能看。

## 7. 聚类阈值回测（余弦 0.82 偏严的怀疑 → 用历史数据调参）

### 目标
当前 `CLUSTER_SIM_THRESHOLD=0.82`（config.py:848）"偏严"的怀疑一直没验证：可能把本应同簇的拆开（召回低），也可能把无关的并错（精度低）。用历史 `scored_latest.json` + 人工标注（feedback.jsonl 的 verdict / 手动簇标签）回测，找到更优阈值。

### 可行性评估
**高（P1，已有基础）**。`backtest.py` 已做"精选阈值回测"，`selection/cluster.py` 已支持 `run_daily_step(threshold=...)` 参数化。缺口：把回测从"精选阈值"扩展到"聚类阈值"，并引入人工标注金标准。

### 技术方案 / 架构草图
```
1. 造金标准（一次性人工劳动）：
   - 取若干历史周次的 scored_latest，人工标"哪些 url 其实讲同一件事"（cluster gold）
   - 小规模即可（如 200 条 / 30 簇），存 data/cluster_gold.jsonl
2. 回测脚本（扩展 backtest.py 或新建 backtest_cluster.py）：
   for thr in [0.70,0.75,0.78,0.80,0.82,0.85,0.88,0.90,0.95]:
       cluster.run_daily_step(thr)  → 得到预测簇
       算 precision / recall / F1 vs gold（同簇判对率）
   输出曲线：阈值 → F1，找出拐点
3. 决策：选 F1 最高的 thr（若 0.82 确实偏严，可能落到 0.78 左右）
4. 护栏：阈值改动需人工确认写入 config；改前 git 备份；保留旧值注释
```
注意：现有聚类是"主规则(entity+event_type) + 余弦回退"，回测主要针对"余弦回退"那段，主规则不动。

### 成本估算
- 人工标注金标准：约 1-2 人日（一次性）。
- 脚本：复用现有 cluster 参数化入口，约 0.5 人日。
- 算力：零。

### 与无后端静态站架构的冲突点
**无冲突**。回测是离线分析，不改生产；最终只动一个 config 常量。

### 建议优先级 & 启动时机
**P1（高）**。建议 2026-07 内做（标注 1-2 天 + 跑回测），直接提升聚类质量（影响去重与推荐理由准确度）。

## 8. 国内商媒一手来源追踪（36氪 / 动脉网 / AgeClub 引用海外源补采）

### 目标
小爽提过的中优先级待办：国内商媒（36氪、动脉网、AgeClub 等）常编译/引用海外一手信源（海外初创、海外报告、海外政策）。识别这些"被引用的海外源"，把原文补进采集，提升一手信息占比与时效。

### 可行性评估
**中（P2）**。技术可行（解析国内文章正文里的外链 / 提到的海外机构名），但难点在"精准识别一手源"和"避免采到营销软文"。价值高（契合"以海外为镜"核心原则）。

### 技术方案 / 架构草图
```
采集增强（build time）：
  1. 国内源（36kr/动脉网/ageclub）文章已进 scored_latest
  2. 新增 extract_first_source.py：
     - 抓原文正文（现有 collector 已有正文抽取能力可复用）
     - 提取 outbound links（正文 <a href> 中非国内域名）
     - 命名实体识别：正文提到的海外机构/公司名 → 与 enterprise 库 / 已知海外源比对
  3. 判定"一手源"：
     - 链接域名在白名单（海外权威媒体/官网/报告站） → 高置信
     - 提到的公司名命中 enterprise 库 overseas → 中置信
  4. 补采：高置信外链 → 加入下周 collector 候选源（或一次性补抓该 URL 进库）
  5. 标注：被补采的文章打 first_source 标签，在展示上标明"源自 X 海外报道"

护栏：
  - 只补采"非营销、非论坛"域名（用现有 SOURCE 层级逻辑复用）
  - 补采量设上限（如每周 ≤20 条），防噪音
  - 误采可经 feedback 标记后进黑名单
```

### 成本估算
- 算力：多抓国内文章正文 + 解析链接，可忽略（已有抽取能力）。
- 维护：海外一手域名白名单需人工维护，约 0.5 人日初始 + 季度更新。
- 价值：高，直接强化"海外为镜"差异化。

### 与无后端静态站架构的冲突点
**无冲突**。补采在 build time 完成，结果烤进 HTML；展示仅多一个标签。

### 建议优先级 & 启动时机
**P2（中）**。建议：事件聚类回测、企业分自进化稳定后（2026-Q3）做；先用手工补 10 条验证链路，再自动化。

## 9. 增量抓取与跨周去重（when:7d 全量重抓 → 增量 + 跨周去重）

### 目标
当前所有 Google News 源用 `when:7d` **每周全量重抓**（config.py:520+），再用 `history["urls"]` 按 url_hash 去重。问题：(1) 7 天窗口导致跨周边界文章被重复处理；(2) 全量重抓浪费请求与重试；(3) 旧文章若不在 7 天窗口内则永远不再见（但也因此不会重复——需权衡）。改为"增量拉取 + 长周期去重"，降负载、降重复。

### 可行性评估
**高（P1）**。去重机制 `history["urls"]` + `url_hash` + `mark_seen`（`collector.py:85-94`）已存在，只是 `history` 的保留期与抓取窗口需调整。Google News 支持 `after:` 日期参数，可改"全量 7d"为"增量 since 上次运行"。

### 技术方案 / 架构草图
```
改动点（全在 collector / config）：
  A. 抓取窗口改为增量：
     - 记录 last_run_date（STATE.md 或 history["last_run"]）
     - Google News query 由 when:7d 改为 after:<last_run_date - 1d>
     - 对 RSS 直连源：用 If-Modified-Since / ETag（requests 自带）跳过未变
  B. 去重窗口拉长：
     - history["urls"] 保留期从"仅当周"延长到 60d（或永久，仅删超期）
     - 跨周边界文章：因 url_hash 已记录，第二次抓到直接跳过，不再重复进 scored
  C. 增量合并：
     - 新文章 append 到 scored_latest（保留历史 N 周用于聚类/回测）
     - 旧文章不打回（避免重复计分），仅当内容更新才刷新
  D. 兜底：每 4 周做一次"全量 7d 校准抓"，防止增量漏抓

护栏：
  - 增量失败（如 last_run 缺失）→ 自动回退全量 7d，不丢数据
  - 去重误杀（同源不同文 url 相似）→ 用 url 精确 hash，不等模糊匹配
```

### 成本估算
- 请求量：增量后每周请求数降约 40-60%（边界文章不重复抓）。
- 算力：history 字典增大（60d），但仍在 MB 级，可忽略。
- 人力：约 1 人日改造 + 回归测试。

### 与无后端静态站架构的冲突点
**无冲突**。纯采集侧优化，产物仍是静态 HTML。

### 建议优先级 & 启动时机
**P1（高）**。建议 2026-07 内做（与死链监控一起，都是采集健壮性）。直接降本、降重复，且为后续"加源"腾出请求预算。

## 10. 死链 / 源健康监控（自动检测挂掉/0 产出源，降权或告警）

### 目标
自动识别：哪些 RSS/代理源已挂（HTTP 错误、长期 0 产出、返回结构异常），自动降权或告警，避免"43 个 Google News 代理源悄悄空转"却无人知（项目全景文档指出 15 个源 0 产出）。

### 可行性评估
**高（P1，已有部分基础）**。`daily_health.py` 已产出信源分布 `top_sources` / `bottom_sources`，`SOURCE_AUDIT.md` 已存在，`loop_audit.py` 已有审计框架。缺口：把"0 产出"从"展示"升级为"检测+降权/告警"，并加 HTTP 可达性探测。

### 技术方案 / 架构草图
```
新增 source_health.py（build time，在 collector 后跑）：
  1. 可达性探测（轻量）：
     - 对每个源发 HEAD/GET（带超时 10s），记录 status / 耗时 / 是否超时
     - Google News 代理源特殊：返回空 feed 也算"可达但无产出"，单独标记
  2. 产出统计（复用 daily_health 的 src_counts）：
     - 连续 N 周（如 3 周）0 产出 → 标记 stale
     - 单周 0 产出但历史有 → 标记 watch
  3. 动作（分级，写 config 或单独 source_health.json，不自动改生产权重）：
     - dead（HTTP 错误/超时连续 2 周）→ 自动降权（ collector 跳过或权重×0.3）
     - stale（0 产出连续 3 周）→ 告警 + 建议人工复核
     - healthy → 正常
  4. 告警渠道：写入 HEALTH_REPORT.md + STATE.md；若接了推送，发一条摘要
  5. 护栏：降权是"软降"（跳过抓取），不删源配置；人工可一键恢复

数据落点：
  data/source_health.json = { source_id: {status, last_ok, consecutive_zero, action} }
  collector 读取它决定本周是否跳过该源
```

### 成本估算
- 算力：44 源各一次 HEAD 探测，可忽略（<1s 级，带超时）。
- 维护：告警复核约 5 分钟/周。
- 价值：高，直接解决"单一失效点 + 空转源"隐患。

### 与无后端静态站架构的冲突点
**无冲突**。监控在 build time 跑，结果进报告/JSON；降权在采集侧生效，不碰静态页。

### 建议优先级 & 启动时机
**P1（高）**。建议 2026-07 内与"增量抓取"同批做（都是采集健壮性），优先于任何"加新源"动作——先修好现有源再扩。

## 0. 先说清楚：现在的数据长什么样

- **企业库（1130 家）**：每家有 `category_l1`（一级行业）、`category_l2`（二级）、`tags`、`value_score`（研究价值分，0–61）、`business_model`、`news_coverage`（含 `news_count` 与 `latest_news` 列表）、`funding_latest` 等。

## 一、赛道全景图 设计

### 1.1 一句话它是啥
把 1130 家企业按"赛道"重新分组，用一个**总览面板**告诉你：哪个赛道企业多、哪个赛道平均研究价值高、哪个赛道最近有动静。让小爽**一眼看出"该去哪个赛道里找选题"**，不用在 1130 张企业卡里瞎翻。

### 1.2 它解决小爽的什么问题
- **没方向感**：现在企业库是平铺的列表，小爽说"看不懂"。全景图先把 1130 家压成十几个赛道，先选赛道、再进赛道挑企业，决策层级清晰。
- **判断"热不热/值不值"**：用"企业数"看赛道储备厚度，用"平均研究价值分"看这个赛道整体值不值得写，用"近期有动态企业数"看最近有没有新鲜由头。
- **快速定位选题方向**：比如"康复与辅助设备"均分最高（28.8）、且近期有 12 家在动 → 优先去看；"长寿科技"虽然企业少（9 家）但全是前沿公司 → 适合做趋势型选题。

### 1.5 纯静态怎么实现（零后端）
1. **赛道映射表写死在生成脚本里**：把 35 个 `category_l1` 归并到上面 14 个赛道（一份 Python dict 常量即可，约 20 行）。

4. **刷新节奏**：跟随每次全站重新生成（已有 daily pipeline），自动更新。

### 1.6 开发量估计

## 二、事件簇对照卡 设计

### 2.1 一句话它是啥
当同一条新闻被多家媒体都报了（系统已把它们聚成一个"簇"），**不堆 N 张重复卡**，而是合成**一张卡**：主条 + "另有 N 家媒体报道" + 各家标题/角度对照。

### 2.2 它解决小爽的什么问题
- **去重、省眼**：现在 63 条精选里已有 5 个多源簇（如"城乡三级养老服务网络指引"被新京报/观点网/大河财立方三家报）。不做合并，小爽会重复看三遍。
- **交叉验证 + 看视角差异**：同一事件不同媒体标题/措辞不同，对照着看能判断"这事到底多大"（3 家都报=真热点）、以及"各家角度差在哪"（有的写政策全文、有的写地方落地、有的写资本反应）——这正是写稿时最有价值的"反常识点/多视角钩子"。
- **辅助判断事件量级**：簇越大（报道家数越多），越值得写；簇小但 `final_score` 高，可能是垂直媒体独家，适合做深挖。

### 2.3 数据上怎么取（已有，零新增采集）
`data/scored_latest.json` 每条已带：
- `cluster_id`：同簇共享同一 ID（如 `C0019`）。
- `cluster_size`：簇内条数（如 3）。
- `is_main`：`true`=该簇主条（挑分数最高/最权威的那条当主条）。
- 同簇其他条带各自 `source`、`title`、`url`、`event_type`。

**取数逻辑（生成时）**：
```
按 cluster_id 分组 →
  若 cluster_size==1：原样出一张普通卡；
  若 cluster_size>1：出一张"簇卡"，主条用 is_main 那条，
    其余条作为"另有 N 家媒体报道"列表，列出 来源 + 标题 + 链接。
```

### 2.4 卡片长什么样（示意）
```
┌─────────────────────────────────────────────┐
│ [融资] 城乡三级养老服务网络建设管理指引发布      │  ← 主条（新京报，final_score 高）
│ 民政部发布…全文/地方落地解读…            [看原文]│
│ ── 另有 2 家媒体报道 ──                      │
│ · 观点网：民政部发布…（全文）          [链接]  │  ← 角度：全文/政策原文
│ · 大河财立方：城乡三级…指引发布        [链接]  │  ← 角度：财经/地方视角
│ 信号强度★ 簇大小 3  │ 最新 2026-07-0x        │
└─────────────────────────────────────────────┘
```
小爽一眼能看：这件事被 3 家报 → 量级够；观点网给全文、大河从财经切入 → 写稿时可借"政策原文 vs 资本反应"做角度。

### 2.5 纯静态怎么实现（零后端）
- 生成脚本（`generator.py` / 资讯页生成逻辑）在读 `scored_latest.json` 时**先按 `cluster_id` 聚合**，簇内 `is_main=true` 的当主卡，其余折叠进"另有 N 家"列表。
- 折叠列表默认收起、点"展开"显示（复用现有卡片交互样式），保持页面不臃肿。
- **完全复用现有聚类产物**，不需要新增任何采集或模型调用。

### 2.6 开发量估计
- **小（约 0.5 天）**。聚类结果已现成，只需在资讯页渲染层加"按 cluster_id 归并 + 折叠列表"的逻辑和样式。无算法、无新数据源。

## 三、竞品固定栏目对标（要不要学 SHN/CB Insights/Longevity.Technology 做"每周固定栏目"）

### 3.1 竞品在做什么（来自《调研_增量信源发现_2026-07-09.md》）
- **Senior Housing News / CB Insights**：固定 **"每周交易汇总"（Deals & Transactions）**——把一周内的并购/融资/资产交易一条流集中放出，扫读效率极高。
- **Longevity.Technology**：**企业数据库 + 每周 "Longevity News Roundup" 周报 + 播客**，"媒体+数据+目录"三位一体。

### 3.2 给小爽的建议（结论先行）
**建议做，但分两步走，不要一上来照搬 SHN 的"纯交易流"。**

**第一步（低成本，强烈建议，本周就能上）：做"每周投融资/事件速览"聚合页。**
- 内容：把一周内 `scored_latest.json` 里 `event_type∈{融资, 并购, IPO, 政策}` 的精选，按时间排成一张"周报"页（可放在规则页 about.html 或独立 `weekly.html`）。
- 价值：小爽周一打开就能看到"上周银发圈发生了啥"，直接当选题池。
- 落地难度：**低**。数据全现成（已有每日精选 + 聚类），只需一个按周聚合的生成函数 + 一个页面模板。**零新采集。**

**第二步（中成本，建议等信源扩完再做）：常态化的"企业库周报 / 赛道动向"栏目。**
- 内容：结合本文第一节的"赛道全景图"，每周输出"哪些赛道本周动态变多"的简短周报。
- 价值：把静态企业库变成"会动的情报"，对标 Longevity.Technology 的周报。
- 落地难度：**中**。需要"赛道×时间"的增量统计——前提是企业库的 `news_coverage` 覆盖度要先提上来（目前只有约 130/1130 家有动态，见附录），否则周报会显得空。建议等《调研_增量信源发现》里那 6 个新 RSS 源接进来、新闻回填更密之后再上。

**不建议立刻照搬的部分：SHN 那种"纯 M&A 交易流"。**
- 理由：SHN 背靠美国养老地产一级市场（交易极度高频），而本站信号源以中文垂直媒体 + Google News 代理为主，**纯交易类一手源目前偏少**（调研里点名的 Ziegler/NIC 等还没接）。硬做交易流会"没货可填"，反而显得空。等养老地产投行源（Ziegler 等）接入后再考虑。

### 3.3 利弊与落地难度一览

| 方案 | 利 | 弊 | 落地难度 |
|------|----|----|----------|
| 每周投融资/事件速览（第一步） | 直接服务选题、复用现有精选、零新采集 | 内容厚度依赖每日精选质量 | **低（~0.5–1天）** |
| 赛道动向周报（第二步） | 对标 Longevity、把静态库变动态情报 | 需先补企业库新闻覆盖度 | **中（~2–3天，且依赖信源扩建）** |
| 照搬 SHN 纯交易流 | 行业范式成熟 | 一手交易源不足，易"开天窗" | 暂不推荐 |

## 附录：数据口径与已知缺口（写文档时如实记录）
1. **赛道均分"0"≠价值为 0**：适老科技·智能硬件·机器人（47 家）、长寿科技与生物医药（9 家）及部分长尾企业**尚未跑 `value_score`**，显示 0 是"未评分"。上线全景图前建议先补评这两批。
2. **"库内有动态"覆盖很薄**：1130 家里仅约 130 家 `news_coverage.news_count>0`，大部分企业暂无新闻挂钩。这会影响"近期有动态企业数"这一列的代表性——它反映的是"库里已挂新闻的企业"，不是"真实活跃企业"。等《调研_增量信源发现》的新源接入、新闻回填变密后，该指标会更准。
3. **聚类已就绪但阈值刚调**：`CLUSTER_SIM_THRESHOLD` 已从 0.82 降到 0.75（config.py:913），能合并绝大多数"同事件不同源"报道；事件簇对照卡直接吃这个产物即可。
4. **所有方案均零后端**：本站是 Python 生成静态 HTML + GitHub Pages，上述三块全部在"生成期"完成聚合，浏览器端只做展示与点击筛选，不涉及任何服务器/数据库。

## 给小爽的一句话决策摘要
- **赛道全景图**：做。低成本（~1天）、直接解决"看不懂企业库"的痛点，先把两个未评分赛道补评即可上线。
- **事件簇对照卡**：做。极低成本（~0.5天）、复用现有聚类，去重+多视角对照，写稿很实用。
- **固定栏目**：先做"每周投融资速览"（低成本、本周可上）；赛道周报等信源扩建后再上；别急着照搬 SHN 纯交易流（货源不够）。

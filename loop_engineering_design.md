## 一、大白话：Loop Engineering 到底是什么？

以前我们跟 AI 打交道是「一次性的」：你下一条指令，它干一次活，完事。
Loop Engineering（循环工程）的核心思想是：**让 AI 自己转圈** —— 干完活 → 检查结果 → 发现问题 → 改自己 → 再干，形成一个不停的闭环。

打个比方：
- 普通模式：你教徒弟做一道菜，他做一次，你尝一口，没了。
- Loop 模式：徒弟做完菜，自己尝、自己记笔记「这次盐放多了」，下次自动少放；还把食客（用户）的评价记下来，慢慢变成大厨。

行业里 2026 年 6 月开始热这个词。吴恩达（Andrew Ng）、Claude Code 负责人 Boris Cherny、OpenClaw 的 Peter Steinberger 都在推。但**反对者也很多**——有人说这不过是 2023 年 AutoGPT 玩剩下的，当年就是因为没有「校验机制」和「边界控制」才失败了。

**关键结论（决定我们怎么做）：**
- 循环必须有**「写」和「查」分离**（一个负责干活，一个负责挑错，互相独立）
- 循环必须有**边界控制**（不能让它无限自调用烧光积分）
- 循环里**最慢的那一层往往最重要**（长期规则进化 > 短期纠错）

## 二、吴恩达三层 Loop（时间尺度不同）

| 层级 | 名字 | 时间尺度 | 负责什么 | 类比到本项目 |
|------|------|---------|----------|------------|
| **L1** | Agentic Coding / 执行循环 | 分钟级 | 干一步→发现错→立刻改 | 采集单源失败→自动换兜底源 |
| **L2** | 系统/质量审计循环 | 天级 | 跑完一轮后审查质量，沉淀规则 | 每天 validator 自检 + 噪音反弹自动收紧阈值 |
| **L3** | 外部反馈循环 | 周/月级 | 真实用户反馈修正产品判断 | 小爽的收藏/导出 → 回灌评分权重 |

**最重要的洞察：L3（最慢）决定质量天花板。** 我们最容易忽略用户反馈回灌，但那才是最有价值的进化。

## 三、我们项目怎么落地（三层映射）

### L1 · 采集容错循环 —— ✅ 已落地（2026-07-08）
- 机制：每个源先试主方法（直连 RSS 或 Google News），失败自动切另一种
- 效果：单源挂了不影响全天采集，不会「全军覆没采到 0 条」
- 对应吴恩达说的「工具调用自优化」

### L2 · 质量审计 + 规则进化循环 —— 🔧 待建（本轮重点）
这是「写查分离」的核心。现有 `validator.py` 只做**检查**，不**进化**。要补上「进化」：

```
每日运行 → generator 生成三页（写）
         → validator 自检（查）：
             ① 噪音嫌疑数 spike？（比昨日突增 → 报警）
             ② 规则漂移？（config 改了但 about 没同步 → 报警）
             ③ 精选数暴跌？（< 阈值 → 报警）
         → 若发现重复≥3次的同类问题 → 自动沉淀到 config「Pitfalls Log」
         → 下次运行自动规避
```

具体要落地的两件事：
1. **噪音 spike 检测 + 自动回调**：连续 2 天噪音 > 阈值 → 自动收紧 `SILVER_WEAK_KEYWORDS` / 提高 `signal_score` 门槛（需谨慎，改前先备份 config）
2. **规则漂移自检**：已有（validator 检测每日/每周一/研究价值/两级相关性描述），保留并增强通知

护栏：L2 进化**只允许改阈值和关键词（零模型成本）**，绝不自动改架构；改动写入 `config.Pitfalls_Log` 留痕，小爽可随时回看。

### L3 · 用户反馈回灌循环 —— 🔧 待建（价值最高）
小爽已经在三页加了 ⭐收藏 + ⬇导出 `feedback.jsonl`（localStorage 持久化）。现在缺「回灌」：

```
小爽导出 feedback.jsonl → 放到 data/ 目录
→ 新自动化「反馈回灌」每周跑一次：
   ① 统计被收藏的选题：哪些赛道/事件类型/地区被偏爱？
   ② 被收藏多的维度（如 novelty 反常识）→ 提高对应权重
   ③ 被收藏少的类型 → 适度降权
   ④ 生成「用户偏好画像」写进 about 或反馈报告
```

护栏：权重调整有上下限（如 ±0.03），不会让某维度失控；调整前备份 config。

## 四、风险与护栏（防止翻车）

调研里反对者最大的担心就是这两个，我们必须规避：

| 风险 | 表现 | 我们的护栏 |
|------|------|-----------|
| **积分失控** | 循环自调用无限烧 token | ① 所有评分用规则兜底，零模型调用；② L2/L3 只在固定时间跑（非实时）；③ 单次运行有步骤数上限 |
| **可观测性丢失** | 后台自己改，人看不懂 | ① 所有自动改动写入 `config.Pitfalls_Log` + `STATE.md`；② 关键改动（阈值/权重）发通知给小爽；③ 改动前自动 git 备份 config |
| **循环崩溃** | 像 AutoGPT 一样跑飞 | ① 写查分离（validator 独立挑错）；② 部署门槛（关键步失败不部署）；③ 单源超时 15s 封顶 |

## 六、一句话总结给小爽

> 我们要让银脉从一个「你让我跑我就跑」的工具，变成一个「跑完会自己挑错、挑完会自己改、改完还参考你收藏口味」的会进化的助手。但进化只在固定时间发生、改动都留痕、绝不偷偷烧积分——你随时能看能拦。

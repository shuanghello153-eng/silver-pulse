## 一、标签是什么、为什么要有标签

- **分类（一级 / 二级）= 互斥骨架**：回答「这家企业**做什么业务**」。一个企业主属 1 个一级 + 1 个二级，稳定、有限枚举。
- **标签（Tag / Facet）= 正交可叠加的细分维度**：回答「它**还有什么特征**」。一个企业可有**多个**标签，灵活、可扩展。
- **双重作用**：
  1. **用户侧**：在分类内/跨分类做二次筛选（如「养老服务」里只看「有 AI科技」且「做认知症」的）。
  2. **机器侧**：后续**评分的二次细分信号**。分类定大盘，标签定细分权重——评分时可用标签叠加加权。
- 一个企业可以有多个标签（已确认）。

## 二、好的标签体系 vs 差的标签体系

**好的标签（保留标准）**
- 正交：维度互不内含（业务模式 / 技术 / 人群疾病 / 融资规模 / 渠道 分开）。
- 不与一/二级分类重名（分类答「业务」，标签答「特征」）。
- 每个标签 ≥ 3 家企业（无幽灵分类）。
- 用用户语言、可理解、可数。
- 受控词表（controlled vocabulary），不随意造词。
- 能被评分引用为加权信号。

**差的标签（已清除 / 禁止）**
- 与分类重名（智慧养老 = 分类、行业媒体 = 分类）。
- 单字业务碎片 / 虚词（护理、照护、生态、礼物、综合养老、平台）。
- 品牌 / 产品名 / 国家地区（ElliQ、海尔、日本、英国）。
- 医疗术语碎片（Medicare、EHR、IoT、AR、VR）。
- 长尾幽灵（仅 1–2 家企业）。
- 混合粒度（把「融资状态」和「产品功能」混为一处）。

## 四、打标签规则（给另一个 AI 执行）

1. **受控词表**：标签只能从本文件第三节的词表取值，禁止自由造词。
2. **禁止项**：国家/地区、品牌/产品名、单字业务碎片、与一/二级分类同名的词、状态碎片（已IPO / 已被收购 / 私营未融资）、虚词（生态 / 平台 / 礼物）。
3. **数量阈值**：每个标签企业数 ≥ 3；不足则合并近义词或删除。
4. **每企业标签数**：1–5 个，优先高信号维度（技术、人群疾病、照护模式），避免过度打标。
5. **赛道词来源**：新赛道词参考《选题库》md 的 `05-赛道` 列；`02-/03-` 优先级标记与「不用看 / Crunchbase截图 / 废弃 / 国家信息待补全」等**内部备注不是标签**，不得引入。
6. **不重复分类**：若某词与一/二级分类同名或语义等同（如 护理服务≈养老服务、养老住宅≈养老地产、智能家居≈智能家居分类、B2C媒体≈行业媒体），**不作为标签**，由分类承载。
7. **操作流程**：对齐词表 → 企业匹配（按 name / name_cn 精确匹配）→ 打标 → 校验（不重复分类 & 计数≥3）→ 写回 `all_enterprises.json`。
8. **变更管控**：新增 / 修改标签必须同步更新本文件与生成器，并经小爽确认，禁止静默引入。

## 〇、为什么反复丢规则？（根因 + 防丢机制）
- **根因**：过去规则散落在对话里、没落成一份"唯一文件"，每次重跑用的映射表只覆盖一部分词，其余被静默丢弃 → SODH / 相亲 / 老年鞋 这类词一次次丢。
- **防丢机制（从此执行）**：
  1. 规则只认这一份文件（SSoT），不再散落对话。
  2. 每次重跑后必须跑"检验清单"（第五节），任何一项不达标就是没过关。
  3. 每次迭代把"这次学到的教训"追加到第六节，不许覆盖历史经验。

---

## 一、三层结构（先分清三个东西）
- **分类（一级/二级分类）**：企业"做什么业务"的固定骨架。**存在库里（category_l1/ category_l2 字段），没删**，只是前台不摆树。
- **标签（一级标签/二级标签）**：企业"还有什么特征"，前台**全量展示、可点筛选**。一个企业可多标签。
  - 一级标签 = 分组容器（约 13 个）。
  - 二级标签 = 最小颗粒的具体词（可直接搜、有区分度）。
- **同类词（扩展词/别名）**：只用于"搜索命中"，**不展示、不单列成标签**。例：搜"老花镜"能命中"眼镜"标签。

---

## 二、好标签的标准（达标才算过关）
1. **最小颗粒度**：二级标签是"能直接搜、有区分度"的最细能力词（认知症 / 临终关怀 / 老年鞋 / 助听器），**不是大伞词**（行业服务 / 消费品 / 文娱社交这种笼统词不能当二级标签）。
2. **一级 ≠ 二级同名**：一级是容器、二级是具体词，两者**绝不能同名**。
3. **不与列表字段重复**：标签不能重复卡片上已有的结构化字段（融资状态 / 被收购 / 地区 / 成立年 等）。这些字段自己会显示，再做成标签就是冗余。
4. **同义词已合并**：意思相同的词并成**一个**规范标签，别名全部塞进"同类词"层（只搜不显）。绝不让同义词各自单列成标签。
5. **每个二级标签 ≥ 3 家企业**：不足 3 家的（幽灵标签）必须并入近义标签或删除。
6. **每个二级标签都有同类词**：至少覆盖常见别名，保证搜得到。
7. **一个二级标签只归一个一级**；每个企业标签数建议 ≤ 5，优先高信号维度。
8. **用用户语言、可数、受控词表**：不随意造词。

## 三、坏标签的标准（出现即必须修）
- 一级=二级同名。
- 企业数 > 100 的伞词（颗粒度太粗嫌疑，必须复查是否该拆/改名/是不是垃圾兜底桶）。
- ≤ 2 家的幽灵标签。
- 没有同类词的标签。
- 与列表字段重复（有融资/融资过亿/已被收购/地区…）。
- 语义错配（把"照护系统/ToB/就业"塞进"行业服务"这类归错组）。
- 噪声词当标签（"废弃""综合养老""反共识""生态""平台""礼物"）。
- 国家/地区、品牌/产品名、单字碎片、医疗术语碎片（Medicare/EHR/IoT/AR/VR）。

## 四、标签怎么来（生成规则）
1. **来源（零丢失，三套不双计）**：一级分类 + 二级分类 + Excel 选题库（细分领域_1 / 细分领域_2 白名单）+ 系统自带标签。凡原始词找不到同义归并的，**保持原值当候选标签，绝不静默丢弃**。
2. **归并**：按"同义词合并"把别名并到规范标签，别名进同类词层。
3. **过滤**：套用第三节"坏标签"清单，删噪声/碎片/幽灵。
4. **产业资本（方案B）**：只保留**自身是投资机构/产业资本方**（VC/PE/CVC/REIT/险企/健康计划/上市产业集团/巨头），与"拿没拿过融资"无关。
5. **落地**：全量重打 → 跑检验清单 → 写回 `all_enterprises.json`。
6. **变更管控**：新增/改标签必须同步本文件 + `_rebuild_tags.py`，并经小爽确认，禁止静默引入。

## 五、检验清单（每次重跑后必须全绿）
| 检查项 | 目标值 |
|---|---|
| 一级=二级同名 | 0 |
| 幽灵标签（<3 家） | 0 |
| 与列表字段重复的标签 | 0 |
| 无同类词的二级标签 | 0（核心标签必须有） |
| 0 标签企业 | 0 |
| 单企业标签数 | ≤ 5 |
| 超大伞词（>100 家） | 逐个复查，非真实类目的兜底桶必须拆/清 |

## 六、迭代经验与反思（只追加，不覆盖）
- **2026-07-12**：发现 v1 的映射表只覆盖 ~46 词导致大量二级分类静默丢弃（SODH/相亲/老年鞋 皆因此丢）。教训：① 规则必须落成 SSoT；② 每次重跑必跑检验清单；③ "最小颗粒度""不与字段重复""同义词合并""一级≠二级"这四条是最常被我遗忘的，单独列为**必查四条**。
- **必查四条（我最容易忘）**：最小颗粒度 / 一级≠二级不同名 / 不与列表字段重复 / 同义词合并到同类词层。

## 七、协作约定（小爽定，长期生效）
1. **UI/UX 小修直接改**：搜索框跳动、placeholder 文案、按钮样式、布局微调等简单问题，AI 直接动手，不用先问、不用出方案。
2. **标签大改动先对齐**：清洗/合并/重命名等涉及数据结构的变化，先给方案文档，小爽确认后再执行；执行后必须交付完整映射大表供核对。
3. **每次改完标签必须发完整结果**：不能只说"校验通过 9/9"。小爽要的是能逐条核对的数据，不是一句结论。
4. **未确定的需求不动**：分类栏是否展示一级标签等"小爽还没点头"的事，默认不动，等明确指示。
5. **命名铁律**：文档/版本用 V1/V2/V3 迭代，禁用"最终版/完整版/最终"；旧版本保留不删。
6. **记忆外置**：反复强调的协作偏好、易忘的要点，AI 必须记到跨项目 `MEMORY.md`，不能每次让小爽重复说。

## 八、标签交付格式（小爽核对用，禁止省略）
每次标签改动后，交付的"完整大表"必须包含以下全部，缺一不可：
- **一级标签列表 + 各自企业数**
- **每个一级下的二级标签 + 各自企业数**
- **每个二级标签的全部同类词（一个不省略）**——小爽要逐条核对颗粒度是否符合"最小颗粒度"要求；写"等 X 词""等 30 词"等于没给，禁止。
- **改动前后对照**：哪些企业从什么标签改成了什么（争议标签如行业媒体必须逐条列）。

> 交付物模板：`output/标签体系_Vx.md`（版本号迭代），含上述全部 + 校验结论 + 改动对照表。
